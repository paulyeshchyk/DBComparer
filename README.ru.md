# DB Compare

Сравнение схем баз данных `MSSQL` ↔ `PostgreSQL` | `MSSQL` ↔ `MSSQL` | `PostgreSQL` ↔ `PostgreSQL`

## Запуск расширения

* В VS Code вызовите диалог команд используя комбинацию клавиш 
  * mac: `⌘ + ⇧ + P` 
  * win: `Ctrl + Shift + P`
* Введите `DB Compare: Open`

## Общее описание

Расширение для Visual Studio Code, которое позволяет сравнить структуру двух баз данных и визуализировать различия в удобном табличном виде.

**Поддерживаются**:

Microsoft SQL Server (MSSQL)

PostgreSQL

**Любая** пара из этих СУБД может быть проанализирована:

`MSSQL` ↔ `MSSQL`

`PostgreSQL` ↔ `PostgreSQL`

`MSSQL` ↔ `PostgreSQL`

Это делает инструмент незаменимым при миграциях, репликации, аудите изменений или синхронизации сред (разработка / тестирование / продакшн).

## Чем это отличается от других решений?

**Нативный подход** – не требует установки сторонних утилит или агентов, всё выполняется внутри VSCode через драйверы `mssql` и `pg`.

**Гибкие фильтры** – вы можете исключить временные таблицы, тестовые схемы или любые другие объекты с помощью регулярных выражений, не пересобирая проект.

**Кэширование** – после первого сравнения метаданные сохраняются локально, повторный запуск происходит за секунды.

**Интерактивный интерфейс** – две схемы отображения (детальный и группированный), цветовая индикация различий (розовый – отсутствие, жёлтый – регистр или несовпадение свойств).

**Работает офлайн** – после загрузки кэша вы можете анализировать результат без доступа к БД.

В отличие от тяжёлых коммерческих решений (Redgate, ApexSQL), это лёгкое, бесплатное расширение, которое решает 90% повседневных задач сравнения схем.

### Маппинг схем

Зачем нужен маппинг схем?

В разных СУБД или даже в разных экземплярах одной СУБД имена схем могут отличаться:

в MSSQL часто используется схема `dbo`,

в PostgreSQL — `public`,

в Oracle-подобных средах — `ora_dbo`.

Если вы сравниваете базы, где одна и та же логическая схема называется по-разному, то без маппинга расширение покажет все объекты как «только в источнике» или «только в приёмнике», хотя фактически они идентичны.

Маппинг схем позволяет задать правила преобразования:

`dbo` → `DBO`

`ora_dbo` → `DBO`

`public` → `PUBLIC`

После применения маппинга объекты с разными именами схем будут сопоставлены как эквивалентные, и вы увидите корректные различия на уровне таблиц, колонок, индексов и процедур.

Правила маппинга сохраняются в `localStorage` и **доступны между сессиями**.

### Нормализация типов

В разных СУБД типы данных называются по-своему, но часто имеют семантические аналоги:

`int` (MSSQL) ↔ `integer` (PostgreSQL)

`varchar(255)` ↔ `character varying(255)`

`bit` ↔ `boolean`

Расширение содержит встроенный механизм приведения типов (не расширяемый пользователем), который нормализует типы к общему виду:

целочисленные → `dword`, `qword`, `word`

строковые → `string(N)`, `fixedstring(N)`, `string(MAX)`

дата/время → `date`, `time`, `datetime`

двоичные → `binary(N)`, `binary(MAX)`

Это позволяет сравнивать типы колонок на семантическом уровне, игнорируя синтаксические различия СУБД.

**Важно**: нормализация включается отдельным чекбоксом. Если вы хотите видеть «сырые» типы (как они хранятся в БД) — отключите её.

### Фильтры (исключение временных объектов)

При работе с большими базами часто встречаются временные таблицы, создаваемые разработчиками или ETL-процессами:

`.*#tmp_123e4567-e89b-12d3-a456-426614174000.*`

`.*_temp_2025_03_15.*`

`.*tbl_Backup_.*`

Эти объекты засоряют результат сравнения и могут занимать минуты на извлечение метаданных.

Фильтры позволяют гибко управлять тем, что вы видите:

Включающие фильтры (OR) – задают список регулярных выражений; объект включается, если его имя соответствует хотя бы одному выражению.

Если в списке только `\*` (звёздочка), то включены все объекты.

Можно указать, например: `^dbo\..\*`, чтобы сравнивать только таблицы из схемы dbo.

Исключающие фильтры (OR) – задают список регулярных выражений; объект исключается, если его имя соответствует хотя бы одному выражению.

Исключение применяется **после** включения.

Пример для временных таблиц regex

```
.*#tmp_.*
.*_temp_
.*tbl_Backup_.*
.*[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$
```

Если вы добавите их в исключающие фильтры, то все объекты с такими именами будут скрыты из результата, и вы увидите только «чистую» схему.

**Совет**: используйте включающие фильтры для быстрого ограничения круга анализируемых объектов (например, только определённые схемы), а исключающие – для удаления известного «мусора».

### Кэширование и экспорт

После первого сравнения метаданные (таблицы, колонки, индексы, процедуры и их параметры) сохраняются в локальный кэш. При повторном запуске (с теми же строками подключения) расширение загружает данные из кэша, что занимает секунды, даже если в базе тысячи объектов.

Все сохранённые кэши отображаются на вкладке «Кэш». Вы можете:

Удалить кэш для конкретной пары БД.

Экспортировать кэш в JSON-файл.

#### Импорт кэшей других пользователей (в планах)

Мы рассматриваем возможность добавления импорта кэшей, созданных другими разработчиками. Это позволит:

Делиться результатами сравнения без доступа к БД (например, для code review).

Загружать эталонные схемы из централизованного хранилища.

Сравнивать свои схемы с «золотым» стандартом, не подключаясь к продакшну.

Эта функция находится на этапе обсуждения и будет добавлена в одном из следующих релизов.

### Заключение

`DB Compare` — это лёгкий, быстрый и гибкий инструмент для сравнения схем баз данных, который решает повседневные задачи разработчиков и администраторов. Он не заменяет полноценные ETL-платформы, но значительно ускоряет анализ различий, миграции и аудит.

Если вы работаете с `MSSQL` и `PostgreSQL`, это расширение сэкономит вам часы ручного сравнения.
